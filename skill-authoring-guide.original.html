核心方法论

六条原则

这六条不是独立的规则，是一套相互咬合的逻辑体。每一条回答设计过程中的一个根本问题。

1
放在哪里 · 持久性梯度
常驻 → 按需 → 参考素材
知识放在哪一层，由它的使用频率决定。每轮都必须在场的东西放进核心 prompt（Soul）；某个角色/场景被激活时才需要的东西放进 skill；在 skill 激活的前提下才按概率取用的深度材料，放进 skill 下的 reference/ 或 scripts/ 文件夹。
判断标准不是"重要不重要"，而是"多久用一次"。一个很重要但只在特定场景需要的规则，不属于 prompt——它属于 skill。放错层级的代价：太高则每轮浪费 context，太低则变成无法触达的死知识。
2
整体质量 · 系统一致性
prompt + skills + scripts 形成统一意图的逻辑体
零件好不等于系统好。所有层（核心 prompt、每个 skill、每个 reference 文件、每个脚本）必须服务于同一个 north star，能形成自洽的推理链。任何一个零件的意图和整体系统矛盾，就会在某个边界产生无法预测的行为。
检验方法：任取系统中的任意一个 skill，能否在它里面找到一句话，把它的存在追溯到核心 prompt 的某个原则？找不到 → 这个 skill 是浮空的，要么补 soul anchor，要么删掉。
3
什么时候建 Skill · Regime 门控
使用者 × 场景 决定是否设计 skill
Skill 的身份由两个维度定义：谁在用它（agent/role）和在什么场景下用（trigger/scene）。任意一个维度变了 → 考虑独立 skill。两个都没变 → 归并、降为 reference 或直接删掉，不新建。多 agent 协作时：设一个 planner skill 负责编排，每个 agent 有自己的独立 skill——不共享 SKILL.md。
最常见的错误：因为"内容相关"就建 skill。正确的问题是："这个使用者和场景的组合，和已有 skill 有没有根本性的不同？"没有 → 别建。
4
单一使用者的完备性 · 正交覆盖
无重叠、无空白、边界显式
对同一个 agent，它的所有 skills 必须满足：任意系统状态恰好触发一个 skill，不多不少。两个 skill 在同一状态都能触发 → 冲突，必须修 trigger。某个状态没有 skill 接管 → 空白，必须填补或明确记录为"故意不覆盖"。每个 skill 结尾用 Not for X → [[sibling]] 显式声明边界——这是接口契约，不是可选项。
5
Trigger 精准 + 意图先行 · 心流
Trigger 是精确入口，SKILL.md 意图在前、方法论在后
Trigger（description 字段）是可观测的明确条件，不是描述性的标签。必须能在任意具体状态下做 yes/no 判断。进入 skill 之后，先说"为什么这个角色存在 / 什么是好的工作"，方法论跟在后面。这产生心流：意图建立方向感，方法论在这个方向下自然涌现——而不是拿着工具找问题。
先写方法论的 skill 让 agent 陷入机械执行；先写意图的 skill 让 agent 带着目的行动，在方法论没有覆盖的情况下也能做出合理判断。信任模型的智能，堵不如疏。
6
Skill 的大小 · 粒度纪律
一个 skill 清晰地说一件事
Skill 的粒度要和它服务的 regime 匹配。太粗：一个 skill 想覆盖所有情况，trigger 变得模糊，内容相互干扰 → 按 regime 拆。太细：一个 skill 只有几行，覆盖一个非常窄的角落 → 合并进父 skill 或降为 reference 文件。正确粒度的标志：读 description 就能清楚知道"在什么情况下用这个"，读完 SKILL.md 之后 agent 知道"在这个 regime 里如何行动"。
测试：把这个 skill 的 description 给不了解系统的人看，他能判断"现在这个情况该不该用它"吗？如果要解释很多背景才能判断 → 粒度有问题，要么 trigger 不准，要么 skill 太粗。
原则 1 展开 · 三个桶

每件知识属于哪一层

按使用频率决定层级，不按重要性。重要但按需才用的东西，不应该常驻 prompt。

Soul / 核心 prompt
常驻，每轮都在
无论哪个角色在运行，这里的内容总是存在于 context 里
放进来的测试：
"这个东西在完全空闲、没有任何任务运行时也必须在场吗？"

放：北极星目标 · 生命周期状态机 · 宪法级约束指针

不放：任何角色专属的方法论、任何场景特定的规则
Skill
按 Regime 挂载
特定角色/场景被激活时才加载；其他时候不在 context 里
放进来的测试：
"这个东西只在某个角色做某件事时才需要吗？"

放：角色存在的意图 · 好工作的判断标准 · 智慧与经验模式 · 边界声明

不放：代码 · 领域事实 · 深度方法论细节（→ reference）
reference / scripts
概率性取用
skill 激活后，agent 在遇到特定子任务时主动读取；并非每次都用
放进来的测试：
"这个东西在 skill 已经激活的前提下，只在某些子情况才需要吗？"

reference/：深度方法论、决策树、路由表、领域事实
scripts/：可执行脚本、工具代码、数据处理

重要性越高、用到概率越高 → 越靠近 SKILL.md 本体（在参考表里排前面）
梯度原则：reference 文件夹内部也有梯度——高频使用的方法论文件放参考表首位，低频的放后面。整个 skill 目录是一个由近到远、由高频到低频的知识漏斗。
原则 3 展开 · Regime 决策矩阵

使用者 × 场景：什么时候建新 Skill

使用者 ↓ · 场景 →
场景变了
场景相同
使用者变了
建 2 个独立 skill
两个维度都变。分别写 skill，各有自己的 trigger、soul anchor、边界声明。永不共享 SKILL.md。
建 2 个独立 skill
场景相同但使用者不同。角色目的、智慧、产出各异——共享 SKILL.md 只会让两个角色都得到错误的上下文。
使用者相同
单 skill 多 mode，或 sub-skill
同一角色，不同场景。先考虑"phase-aware"单 skill（多模式）；如果场景差异够大才拆 sub-skill。
不建新 skill
这是补充材料，不是新 regime。归并到现有 skill 的 reference/ 或直接删掉。建了只会产生 trigger 冲突。
多角色协作时：设一个 planner skill（负责编排、路由、生命周期）+ N 个 agent sub-skills（各自的意图和方法论）。Planner 只管调度，不管执行。所有 agent 共享的宪法规则 → Soul，不重复写进每个 sub-skill。
原则 5 展开 · SKILL.md 解剖

SKILL.md 的结构：意图在前，方法论在后

SKILL.md 只放文字和方法论。代码 → scripts/，决策树 → reference/，领域事实 → materials/。

必须
Trigger
description: Use when…
可观测的明确入口条件。在任意具体状态做 yes/no 判断。必须含排除声明：Not for X (→ [[sibling]])。这是机器可读的契约，不是描述文字。
必须
Soul anchor
**Soul anchor:** `id`
追溯到核心 prompt 里的具体原则。没有这行，skill 是浮空的——审阅者和 agent 都无法验证系统是否自洽。
必须
意图
## 存在的目的
1–3 句。这个角色为什么存在。写成推理质量的标准，不是步骤列表。这是 skill 的灵魂——agent 从这里建立方向感，然后方法论才有意义。
必须
边界声明
## Not for
显式排除：→ 不做 X → [[兄弟 skill]]。这是接口契约。没有它，正交性无从验证，scope 悄悄蔓延。
可选
好工作标准
## 什么是好的工作
从外部可观察的结果质量描述成功。让 agent 能自评，不需要人来判断。写成"好工作应该是……"，不是"完成了哪些步骤"。
可选
智慧与经验
## 经验与智慧
积累的模式和陷阱。必须标注"参考，不是约束"。宪法级不变量（write≠call, freeze-before-results…）单独分节、明确标注为法律。两者不能混写。
可选
参考表
## 工具与参考
场景 → 文件路径。纯文字，无代码。每个列出的文件必须存在且有内容。Skill 是索引，深度在 reference/ 里。高频在前，低频在后。
常见问题诊断

Skill 的六种病态

Trigger 模糊
description 能在多个不同状态下触发，无法做确定性判断。
把 trigger 改成可观测条件，加排除声明。每个具体状态的答案必须是明确的 yes 或 no。
Trigger 冲突
两个 skill 在同一系统状态下都可以触发，agent 面对两个权威。
画状态图，确保任意状态最多触发一个 skill。在双方 description 里加互相排除的声明。
两个角色共享一个 SKILL.md
两个不同 agent 用同一个 skill 文件，意图、智慧、产出都不一样但被迫合并。
按角色拆分。共用的方法论 → reference 文件。共用的规则 → Soul 或宪法 skill。
层级放错
跨角色的宪法规则藏进了某一个角色的 L1 skill（其他角色看不见）。或角色专属的内容进了 Soul（每轮浪费）。
所有角色都需要 → Soul。只有这个角色需要 → L1 skill。只在特定子任务需要 → reference。
SKILL.md 里有代码
skill 文件包含实现代码、决策树、技术 schema。方法论意图被技术细节淹没，skill 变得脆弱（代码一变 skill 就要改）。
代码 → scripts/。决策树 → reference/routing-table.md。Schema → reference/。SKILL.md 只放文字。
覆盖空白
系统存在某个真实的状态，但没有 skill 负责处理它。Agent 进入这个状态后无依据，只能即兴发挥。
列出所有有意义的系统状态，为每个状态找到 skill owner。空白要么填补，要么明确记录为"故意不覆盖"。
发布前自查

七个问题

1
每个 skill 对应唯一的 (使用者, 场景) 组合吗？
如果两个 skill 服务相同的 agent 和相同的 trigger 条件，合并它们。Regime 相同 → 不需要两个 skill。
2
所有 skill 能追溯到 Soul 里的具体原则吗？
每个 skill 能完成："这个 skill 服务于 Soul 原则 [X]，因为……"。完成不了 → 要么 Soul 缺这个节点，要么 skill 不属于这个系统。
3
任意系统状态最多触发一个 skill 吗？
把所有 skill 的 description 放一起看，找交集。有交集 → trigger 冲突，修排除声明。完整状态空间覆盖了吗？有空白 → 补 skill 或明确记录。
4
SKILL.md 里有没有代码或技术实现细节？
扫描代码块、决策树、数据 schema。发现了 → 移到 scripts/ 或 reference/。SKILL.md 是方法论文档，不是实现文档。
5
所有 reference 文件都有 skill 指向它们吗？
在所有 SKILL.md 里搜索每个 reference 文件名。零命中 = 孤儿文件，agent 永远找不到它。加进某个 skill 的参考表，或者删掉。
6
智慧（参考）和法律（不可违反）分开标注了吗？
宪法级约束（write≠call, freeze-before-results…）必须标注为不可违反。经验模式必须标注"参考，不是约束"。混在一起 → agent 不知道哪个可以灵活处理，哪个是红线。
7
每个 skill 的粒度是否刚好说清一件事？
读 description，不了解系统的人能判断"现在该不该用它"吗？需要很多解释 → trigger 不准或粒度太粗。不到 10 行内容 → 考虑降为 reference 文件。

Further Reading

以下两个话题不是核心原则，但在设计成熟的系统时值得考虑。

输出契约（Output Contract）
每个 skill 应该明确它产出什么——artifact 名称、文件路径、格式约定。上下游通过固定产物通信，而不是直接耦合。这让 skills 可以独立演化而不破坏整体。设计好了，你可以换掉任何一个 skill 的实现，只要它产出相同的 artifact。
→ 参考：固定路径接线约定，round_result.json 等模板
可演化性（Evolvability）
不是所有 skill 都应该能被 agent 修改。区分"锁死"（Constitutional/Locked）和"可演化"（Seed）：锁死的是宪法级约束和评估权威，agent 可以读但不能改；可演化的是方法论 skill，agent 在使用中可以改进。这决定了系统的信任模型。
→ 参考：zihua pack 的 locked/ vs seed 分块设计
